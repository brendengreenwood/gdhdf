Design × Agents

The Agent-Native
Design Stack

What it looks like when your design system is built for machines to use — not just humans.

The Problem
Current State

Agents guess at your design system

The model generates a component from memory.

Wrong token values.
Deprecated components.
Hardcoded hex colors.

Compliance enforced after the fact — if at all. Design system adoption requires human discipline.

Better Model

Agents query before they generate

The agent calls into the design system first.

✓ Real token values.
✓ Current component APIs.
✓ Usage rules and rationale.

Non-compliant output becomes structurally harder to produce. Compliance is architectural, not disciplinary.

Pipeline
Sprout MCP — What It Exposes
get_token(name)
Returns resolved token value — semantic or raw. Agent uses real values instead of guessing.
{ "name": "color.primary",
  "value": "#00843d",
  "type": "semantic" }
list_components()
Returns all current components with status flags — active, deprecated, experimental.
[ { "name": "DataTable",
    "status": "active" },
  { "name": "OldGrid",
    "status": "deprecated" } ]
get_usage(component)
Returns prop types, required fields, usage examples, and known anti-patterns.
{ "props": [...],
  "examples": [...],
  "avoid": [...] }
get_rationale(id)
Why this token or pattern exists. Helps agents make judgment calls that respect the system's intent.
{ "rationale": "Maintains
  4.5:1 contrast across
  all surface types" }
validate_markup(tree)
Pass a component tree, get back compliance issues before anything ships.
{ "issues": [{
  "type": "hardcoded-color",
  "node": "div.header" }] }
suggest_component(intent)
Describe what you're building. Returns the closest Sprout match with confidence score.
{ "match": "StatusBadge",
  "confidence": 0.91,
  "alt": "Chip" }
Paper — The Execution Surface
Code-native canvas.

What you design is actual HTML/CSS. Nothing gets lost in translation.

What agents can read
Context
  • Existing frames and layouts
  • Current styles and token bindings
  • Component structure as JSX
  • Tailwind output of any node
  • Screenshots for visual context
What agents can write
Generation
  • Create new frames from scratch
  • Update styles on existing elements
  • Set text content and copy
  • Generate responsive variants
  • Apply Sprout tokens directly
Why it matters
The difference
  • Output is real code, not a mockup
  • No handoff — canvas to codebase directly
  • Agent work can be reviewed visually
  • Figma AI sees layer names. Paper sees the web.
How It Works Together
01 — Read skill
Agent reads SKILL.md
Learns when to query Sprout, how to interpret output, what deprecated flags mean, how to handle no-match.
02 — Query system
Agent queries Sprout MCP
Lists components, fetches tokens, gets usage rules for the specific pattern it needs to build.
03 — Generate
Agent writes to Paper
Creates the component grounded in real Sprout token values and component APIs. Output is HTML/CSS, not a Figma layer.
04 — Validate
Agent self-validates
Calls validate_markup() before finishing. Catches compliance issues without a human in the loop.

The key insight: Design system compliance stops being a discipline problem. It becomes an architectural property of how agents are wired up. The agent can't easily generate non-compliant UI if it's querying Sprout first.

vs. Claude in Figma
Claude in Figma

A prompt box that knows your layer names

  • Figma's proprietary format — not real code
  • No design system awareness by default
  • Generates from model memory — wrong tokens, hallucinated props
  • Output still requires handoff — translation loss inevitable
  • Compliance enforced by humans after the fact
  • Limited to whatever Figma chooses to expose
vs
Sprout MCP + Paper

An agent that queries before it generates

  • Real HTML/CSS — what you see is what ships
  • Grounded in your actual design system at query time
  • Token values, component APIs, usage rules all verified
  • Canvas to codebase directly — no translation step
  • Compliance architectural, not disciplinary
  • You own the interface layer — MCP, CLI, SKILL.md
Build Order

Start with the core.
Everything else is a wrapper.

01
Core library — query logic for Sprout tokens, components, rules. This is the real work.
02
CLI wrapper — thin shell interface. Good for pipelines, audits, CI.
03
MCP server — thin MCP interface over the same core. Same logic, different transport.
04
SKILL.md — tells agents when and how. Documentation that agents actually read.
05
Paper integration — point agents at Paper as the canvas. The execution surface closes the loop.
